# FlexibilityEvent Microservice

## 1. Overview

The FlexibilityEvent microservice analyses incoming telemetry and generates flexibility offers/commands for the Prosumers.
- Evaluates telemetry and emits Flexibility Offers when assets meet configured conditions (for example: battery SoC > 90% during peak hours).
- Persists generated events in MySQL using Quarkus reactive client and Mutiny.
- Publishes flexibility offers to a Kafka topic for downstream consumption.

This service represents commands or offers generated by the VPPaaS. For example, if the grid is stressed, the system emits an event offering a financial incentive for prosumers to discharge energy.

## 2. API Endpoints

Base path: `/FlexibilityEvent`

FlexibilityEvent endpoints:
- `GET /FlexibilityEvent`
- `GET /FlexibilityEvent/{id}`
- `GET /FlexibilityEvent/asset/{assetId}`
- `GET /FlexibilityEvent/type/{eventType}`
- `POST /FlexibilityEvent/evaluate/{prosumerId}`

The `POST /FlexibilityEvent/evaluate/{prosumerId}` endpoint accepts telemetry (JSON) and returns:
- `200` with the created `FlexibilityEvent` when an event is generated.
- `204` when telemetry does not trigger any event.

Swagger UI available at:
- `http://localhost:8084/q/swagger-ui`

### 4.2 Covered Functional Scenarios

FlexibilityEvent scenarios:
- Get all flexibility events.
- Get event by id (found and not found).
- Get events by asset id (non-empty and empty responses).
- Get events by event type.
- Evaluate telemetry to create events:
  - High SoC (e.g., > 90%) -> `ARBITRAGE_SELL` with `recommendedAction=DISCHARGE` and incentive calculation.
  - Low SoC (e.g., < 20%) -> `BALANCING_UNAVAILABLE` with `recommendedAction = UNAVAILABLE`.
  - Normal SoC -> no event (204).

### 4.3 What Is Validated

- REST behavior (status codes and response structure).
- Domain mapping from DB result to API responses.
- Correct handling of success, not-found, and no-content outcomes.
- Event persistence and returned inserted id behavior.
- Kafka message publishing when an event is created.

## 8. Implementation Notes

- Event types implemented in code include `ARBITRAGE_SELL` and `BALANCING_UNAVAILABLE`.
- `ARBITRAGE_SELL` is generated when `TelemetryDTO.State_of_Charge > 90.0` and sets `recommendedAction = "DISCHARGE"`.
- `BALANCING_UNAVAILABLE` is generated when `TelemetryDTO.State_of_Charge < 20.0` and sets `recommendedAction = "UNAVAILABLE"`.
- Market price is currently returned by `getCurrentMarketPrice()` as a fixed value `150.0f` for demo purposes.
- Incentive is calculated by `calculateIncentive(soc)` returning `(soc - 90.0f) * 2.0f` for high SoC offers.
- Generated events are persisted in a `FlexibilityEvent` table and the inserted id is returned to the client and included in the emitted Kafka message.
- Kafka topic used by the service: `flexibility-offers` (configured via the reactive messaging channel name).
